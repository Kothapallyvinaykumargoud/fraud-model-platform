# Decision Log — Fraud Model Platform Brief

## 2026-07-20

- **Scope boundary**: Project starts after model approval (ML Platform side only). Data Science / model research is out of scope in spirit, but the user will train a minimal model themselves to have something real to deploy — effort there should stay small.
- **Stakes**: Personal/portfolio learning project. Not production, not a pitch. Right-sized: no investor-grade rigor.
- **Model provenance**: No existing model. User will train a simple fraud detection model on Kaggle's "Credit Card Fraud Detection" (ULB) dataset — confirmed 2026-07-20.
- **Compute**: Self-managed Kubernetes (k3s or kubeadm) on a single AWS free-tier EC2 instance (t2.micro/t3.micro, 1 vCPU, 1GB RAM). Explicitly rejected managed EKS to avoid the ~$0.10/hr control-plane cost. This is a firm constraint.
- **CI/CD**: Rejected self-hosted Jenkins after resource-budget review (750 free-tier EC2 hours/month cover one instance running 24/7, not two, and Jenkins + k3s + model server would not fit in 1GB RAM together). Chose GitHub Actions instead — builds the Docker image and triggers deployment to the EC2 k3s box. Deviates from the original diagram's "Jenkins CI/CD Pipeline" box; user explicitly traded Jenkins hands-on experience for a workable free-tier architecture.
- **Monitoring & Logging**: Self-hosted Prometheus + Grafana as pods on the same k3s cluster, chosen over lightweight CloudWatch, for deeper hands-on ops learning — despite adding RAM pressure to an already-tight 1GB box. Flagged as a real risk (see brief Risks section), not resolved yet — may need short metric retention, disabled kube-state-metrics, or dropping to node-exporter + Grafana only if the box can't hold it all.
- **API consumer ("Banking Applications")**: A self-built script/client (not a UI) that sends realistic transaction payloads to the inference API and prints the fraud verdict — doubles as backend client/contract-design practice.
- **Skill framing (corrected)**: Explicit user instruction — this is a **backend engineering project applied to MLOps**, not generic backend work with the ML stripped out. "Validate Model," "Package Model," and "Rollback" are real MLOps concerns, not generic deploy steps: they should involve actual model versioning/packaging discipline (e.g. a simple model registry — even a versioned S3 prefix or tagged artifact — validation gates before promotion, and rollback that targets a specific model version, not just a container image). Monitoring should include model-level signals (prediction distribution, latency, class balance of predictions) alongside standard infra metrics (CPU/RAM/pod health). This governs scope and success criteria throughout.
- **Timeline**: Open-ended learning, no fixed deadline.
- **Skill level**: New to Kubernetes and Jenkins (Jenkins now moot — replaced by GitHub Actions). Docker comfort not explicitly confirmed — treated as workable but not assumed expert.

- **Model registry**: MLflow chosen over a plain S3-prefix registry for full tracking/registry/stage-transition experience. To fit the 1GB RAM budget, MLflow runs **on-demand only** (locally, or started/stopped on the EC2 box during registration), not as an always-on k3s pod — same on-demand pattern already applied to CI. The serving path reads a "current production model" pointer rather than depending on MLflow being up.

- **Data drift + automated retraining (added scope)**: User explicitly expanded scope beyond the original diagram to include data drift detection and automatic retraining on new data — this is now a core capability, not a stretch goal.
  - **New data source**: Simulated drift from held-out data — a slice of the training dataset is withheld and fed in over time (optionally with injected synthetic shifts, e.g. altered amount/time distributions) to produce reproducible, controllable drift rather than depending on a real live feed.
  - **Where it runs**: On the EC2 box, on-demand — drift check + retraining triggered manually/on-schedule on the same instance as MLflow (not GitHub Actions), consistent with the "on-demand, not always-on" pattern, but this stacks with MLflow's on-demand runs as more load sharing the same 1GB box during those windows (steady-state serving load unaffected since neither runs continuously).
  - **Flow**: drift check -> (if triggered) retrain -> validate -> register new version in MLflow -> CI (GitHub Actions) picks up and redeploys to k3s.

## Open / unresolved (carried into brief Risks section)

- Whether Prometheus + Grafana + k3s + inference API actually fit in 1GB RAM without heavy tuning — unverified until built.
- Whether the user's AWS account is still within its 12-month free-tier window (free tier is time-limited per account, not permanent) — unconfirmed.
- Exact shape of the "model registry" (S3 prefix vs. something more formal like MLflow — MLflow itself would add more RAM pressure to the box) is not yet decided.
